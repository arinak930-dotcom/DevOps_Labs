## Плохие практики написания Dockerfiles

### 1. Использование sudo

```
RUN sudo apt-get update && apt-get install -y cowsay fortune
```

**Что плохо:**  
Приводит к непредсказуемому поведению + надоедает постоянный ввод админского пользователя

**Исправление:**  
Убрать sudo

```
RUN apt-get update && apt-get install -y cowsay fortune 
```

**Что меняется:**  
Экономия места, упрощение кода


### 2. Неиспользование -y

```
RUN apt-get update && apt-get install cowsay fortune
```

**Что плохо:**  
Без использования -y сборка докерфайла может закончиться ошибкой без вмешательства человека

**Исправление:**  
Добавили -y после install

```
RUN apt-get update && apt-get install -y cowsay fortune
```

**Что меняется:**  
Сборка не зависнет в ожидании пользователя, выполнится автоматически


### 3. Использование ADD вместо COPY

```
ADD entrypoint.sh /entrypoint.sh
```

**Что плохо:**  
В зависимости от расширения файла может произойти либо копирование, либо распаковка, а в нашем случае требуется именно копирование

**Исправление:**  
Меняем ADD на COPY

```
COPY entrypoint.sh /entrypoint.sh
```

**Что меняется:**  
Предсказуемо произойдет копирование файла


## Плохие практики при работе с контейнерами

### 1. Создание долгосрочных контейнеров

Образ, определенный в Dockerfile, не должен создавать контейнеры, которые могут функционировать долгосрочно. То есть контейнер не должен быть таким, что его нельзя будет разрушить без возможности перестройки и замены с минимальным набором настроек и конфигураций. Сюда же относится плохая практика загрузки в контейнер данных, которые больше нигде не фигурируют, так как контейнер в любой момент можно удалить и данные потеряются.

### 2. Использование одного контейнера для нескольких задач

Например, нельзя в одном контейнере запускать сразу 2 приложения, необходимо для каждого приложения создать отдельный контейнер. Это связано с тем, что при работе 2-х приложений в 1-м контейнере между приложения могут возникать конфликты, поэтому важно разделять их работу.

## Плохие практики написания Docker compose файлов

### 1. Задавать фиксированное имя контейнеру

```
container_name: cow
```

**Что плохо:**  
Может вызывать конфликты с другими именами контейнеров; мешает масштабированию, т.к. нельзя в таком случае контейнеры назвать одинаково

**Исправление:**  
Убираем это.

**Что меняется:**  
Уменьшаем конфликты.


### 2. Использовать restart: always

```
restart: always
```

**Что плохо:**  
Нерелевантно использовать в данном случае, т.к. у нас простое приложение; если приложение падает из-за ошибки, то оно будет подниматься все время из-за restart: always

**Исправление:**  
Убираем это.

**Что меняется:**  
Уменьшаем конфликты.


## Настройка изоляции контейнеров

В compose-проекте были созданы два сервиса, cowsay_correct1 и cowsay_correct2, которые собираются из одного Dockerfile. Для сетевой изоляции каждому сервису была назначена собственная сеть. Теперь контейнеры запускаются в одном проекте одной командой docker compose up, но не могут взаимодействовать друг с другом по сети, так как у них отсутствует общая сеть.
