## Плохие практики написания Dockerfiles

### 1. Использование sudo

```bash
docker build -t my-image .
docker run my-image
```

**Что плохо:**  
приводит к непредсказуемому поведению + надоедает постоянный ввод админского пользователя

**Исправление:**  
убрать sudo

**Что меняется:**  
экономия места, упрощение кода

---

### 2. Неиспользование -y

**Что плохо:**  
без использования -y сборка докерфайла может закончить ошибкой без вмешательства человека

**Исправление:**  
добавили -y после install

**Что меняется:**  
сборка не зависнет в ожидании пользователя, выполнится автоматически

---

### 3. Использование ADD вместо COPY

**Что плохо:**  
в зависимости от расширения файла может произойти либо копирование, либо распаковка, а в нашем случае требуется именно копирование

**Исправление:**  
меняем ADD на COPY

**Что меняется:**  
предсказуемо произойдет копирование файла

---

## Плохие практики при работе с контейнерами

### 1. Создание долгосрочных контейнеров

Образ, определенный в Dockerfile, не должен создавать контейнеры, которые могут функционировать долгосрочно. То есть контейнер не должен быть таким, что его нельзя будет разрушить без возможности перестройки и замены с минимальным набором настроек и конфигураций. Сюда же относится плохая практика загрузки в контейнер данных, которые больше нигде не фигурируют, так как контейнер в любой момент можно удалить и данные потеряются.

---

### 2. Использование одного контейнера для нескольких задач

Например, нельзя в одном контейнере запускать сразу 2 приложения, необходимо для каждого приложения создать отдельный контейнер. Это связано с тем, что при работе 2-х приложений в 1-м контейнере между приложения могут возникать конфликты, поэтому важно разделять их работу.
